# Dev Note 작성 스킬

args = `{{ args }}`

현재 대화에서 논의된 작업 내용을 `.claude/.dev_note/` 패치 노트 파일로 정리합니다.

---

## STEP 1 — 기존 파일 파악

1. Glob으로 `.claude/.dev_note/*.md` 파일 목록을 가져옵니다
2. 가장 최근 파일(파일명 기준 내림차순 정렬의 첫 번째)을 Read하여 포맷을 확인합니다
3. 기존 파일명에서 버전 패턴을 추출합니다:
   - `1.0.12_ver.md` → 메인 버전 파일 패턴: `<major>.<minor>.<patch>_ver.md`
   - `1.0.11_hotfix_1.md` → 핫픽스 패턴: `<ver>_hotfix_<n>.md`
   - `1.0.11_bugfix_1.md` → 버그픽스 패턴: `<ver>_bugfix_<n>.md`
   - `1.0.11_user_checkbox.md` → 기능별 패턴: `<ver>_<slug>.md`

---

## STEP 2 — 작업 내용 분석 (`Agent` 툴, model: **opus**)

현재 대화의 요구사항을 빠짐없이 추출하기 위해 **Opus 서브에이전트를 호출합니다.**

### 2.1 호출 전 준비

Agent 프롬프트에 포함할 대화 요약을 직접 파악합니다:
- 사용자의 구현 요청 내용 (핵심 문장 또는 요약)
- 논의된 기술적 결정 (파일명, 함수명, 엔드포인트, 알고리즘 등)
- 이미 합의된 설계 방향
- 아직 결론나지 않은 사항

### 2.2 Opus 서브에이전트 호출

`Agent` 툴을 실행합니다. `model: "opus"`를 명시합니다.

서브에이전트에게 다음 내용을 프롬프트로 전달합니다:

- **대화 컨텍스트**: 2.1에서 파악한 내용 전체

- **작업 유형 판단** (하나를 결정하도록 지시):

  | 작업 유형 | 파일명 패턴 예시 | 사용할 스킬 |
  |----------|----------------|-----------|
  | 신규 기능 / 복합 개선 (새 파일·추상화·의존성 포함) | `1.0.12_ver.md` 또는 `1.0.12_<slug>.md` | `feature-dev` |
  | 버그 수정 (원인 분석 포함, 중간 규모) | `1.0.11_bugfix_<n>.md` | `feature-dev` |
  | 소규모 수정 (단일 관심사, 50줄 미만, 새 추상화 없음) | `1.0.11_hotfix_<n>.md` | `micro-fix` |
  | 단일 기능 개선 (중간 복잡도) | `1.0.11_<slug>.md` | 판단에 따라 결정 |

- **추출 항목** (각 항목을 빠짐없이 채우도록 지시):
  - **작업 제목**: 한 줄 (한국어, 동사 명사형으로 끝냄 — 예: "역할 import 상한 환경변수화")
  - **작업 배경/동기**: 왜 필요한가
  - **구체적 수정 내용**: 대화에서 명시된 파일 경로·함수명·로직 변경 사항
  - **암묵적 요구사항**: 명시되지 않았지만 이 작업에 포함돼야 할 것들
    - 엣지 케이스 (빈 배열, null, 권한 없는 사용자, 동시 요청 등)
    - 예외 처리 (실패 시 어떻게 동작해야 하는가)
    - 경계 조건 (0개·1개·N개 시나리오)
  - **수정 대상 파일**: 변경이 예상되는 파일 목록
  - **테스트 시나리오**: 검증 방법
  - **미확정 사항**: 대화에서 결론이 나지 않은 것 (TODO로 표시)

서브에이전트 결과를 받은 후 STEP 3으로 진행합니다.

---

## STEP 3 — 파일명 제안 및 확인

파악한 정보를 바탕으로 파일명 후보 1-2개를 제안합니다:

```
파일명 제안:
1. .claude/.dev_note/1.0.12_<slug>.md  (기능별 패턴)
2. .claude/.dev_note/1.0.12_ver.md     (버전 파일로 통합)

어떤 파일명을 사용할까요? (번호 선택 또는 직접 입력)
```

**사용자 확인을 받은 후** STEP 4로 진행합니다.

---

## STEP 4 — 파일 작성

확정된 파일명으로 아래 템플릿 구조에 맞춰 Write 툴로 파일을 생성합니다.

### 파일 구조 템플릿

```markdown
# <버전> <작업 제목>

---

## (사용할 skills: <feature-dev 또는 micro-fix>)

## 개요

<작업 배경과 목적을 2-5문장으로 설명>

---

## 영향도 분석

### 이 기능이 변경하는 상태
- (예: AwOrganizationMember 레코드 삭제, 권한 배열 수정 등)

### 그 상태를 전제로 동작하는 기존 로직
- (예: AwRbacService Rule 3 — no-membership 조건 분기)
- 없으면: 없음

### delete / remove / revoke / bulk 계열이면 추가 확인
- 이 기능이 "없음(0개)" 상태를 만들 수 있는가?
- 기존 로직 중 해당 상태를 "초기 상태"로 가정하는 코드가 있는가?

---

## 수정 내용

### 1. `<파일 경로>` — <변경 요약>

<구체적인 변경 내용 설명. 대화에서 논의된 코드 스니펫이 있으면 포함>

### 2. `<파일 경로>` — <변경 요약>

<...>

---

## 수정 대상 파일

- `<파일 경로 1>`
- `<파일 경로 2>`

---

## 테스트

- <테스트 명령어 또는 시나리오>
- <추가 검증 방법>
```

### 작성 원칙

- 대화에서 이미 결정된 내용은 그대로 반영합니다 (추측하지 않음)
- 미확정 사항은 `TODO:` 또는 `<결정 필요>` 로 표시합니다
- 코드 스니펫은 대화에서 명시적으로 논의된 것만 포함합니다
- 수정 내용 섹션은 실제 논의된 파일/함수 기준으로 작성합니다

---

## STEP 5 — 작성 완료 보고

파일 생성 후 다음 형식으로 보고합니다:

```
dev_note 작성 완료: .claude/.dev_note/<파일명>

사용할 스킬: <feature-dev / micro-fix>
작업 항목: <n>개

이제 /devnote 로 파일을 로드하고 진행하시겠습니까?
```
