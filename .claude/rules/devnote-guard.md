# Dev Note 가드 규칙

이 프로젝트(igloo_workflow)에서 **코드·구성·문서를 수정하는 모든 구현 요청**은 `.claude/.dev_note/` 패치 노트를 통해 진행하는 것을 원칙으로 합니다.

본 룰의 목적은, 사용자가 무의식적으로 날려버린 구현 요청을 그냥 받아서 바로 진행하지 않고, 작업을 계획 파일(dev_note)로 남길 기회를 제공하는 것입니다. **마찰을 만드는 것이지, 하드 블록이 아닙니다.** 사용자가 "그냥 진행해줘"라고 하면 바로 진행합니다.

---

## 1. 가드 트리거 조건

사용자 메시지에 아래 동사·표현이 포함되고, **2번 예외 조건**에 해당하지 않으면 가드를 발동합니다.

### 트리거 동사 (한국어)
- "고쳐줘", "수정해줘", "변경해줘", "바꿔줘"
- "추가해줘", "만들어줘", "넣어줘", "붙여줘"
- "구현해줘", "개발해줘", "작업해줘"
- "삭제해줘", "제거해줘", "빼줘"
- "리팩토링해줘", "정리해줘"

### 트리거 동사 (영어)
- "fix", "patch", "correct"
- "add", "implement", "create", "build"
- "remove", "delete"
- "refactor", "clean up", "update"

---

## 2. 예외 조건 (가드 발동 안 함)

다음 중 하나라도 해당하면 가드를 건너뛰고 바로 진행합니다:

1. **devnote 스킬 경유**: 현재 대화에서 `/devnote` 스킬이 이미 실행되어 파일이 로드됐거나, args에 `devnote:<파일명>` 형식이 포함된 경우
2. **명시적 바이패스**: 사용자가 이미 "그냥 진행해줘", "dev_note 없이 해줘", "바로 해줘" 등으로 가드 확인에 답변한 경우
3. **비구현 작업**: 질문, 코드 설명, 분석, 리뷰 요청, 파일 읽기, 현황 파악 등 코드를 직접 변경하지 않는 작업
4. **rules/skills/CLAUDE.md 수정**: `.claude/` 하위 파일만 수정하는 작업 (dev_note 작성, 스킬 수정 등 포함)
5. **자명한 소규모 수정**: 오타 수정, 줄 1-2개 단순 수정처럼 dev_note 작성 비용이 수정 비용보다 명확히 큰 경우

---

## 3. 가드 발동 시 행동

트리거 조건에 해당하면, **구현을 시작하기 전에** 다음 형식으로 사용자에게 확인합니다:

```
dev_note를 통한 작업이 아닙니다. 어떻게 진행할까요?

1. 그냥 진행 — dev_note 없이 바로 구현합니다
2. dev_note 먼저 작성 — /devnote-write로 계획 파일을 작성한 뒤 진행합니다

(번호 또는 원하는 방식을 말씀해주세요)
```

---

## 4. 사용자 응답에 따른 행동

| 사용자 응답 | 행동 |
|-----------|------|
| "1" / "그냥 진행해줘" / "바로 해줘" 등 | 즉시 feature-dev 또는 micro-fix 스킬로 구현 시작 |
| "2" / "devnote 쓰자" / "dev_note 먼저" 등 | `/devnote-write` 스킬을 호출 — 현재 대화 맥락(요청 내용 + 논의된 방향)을 args로 전달 |
| 추가 질문이나 다른 내용 | 답변 후 다시 선택지를 제시 |

---

## 5. Auto mode 주의

**Auto mode가 활성화된 상태에서도 이 가드는 반드시 작동합니다.**

Auto mode는 세부 구현 결정을 위임한 것이지, 계획 없이 대형 작업을 시작하도록 허용한 것이 아닙니다. 예외 조건 5번(자명한 소규모 수정)에 해당하는 경우는 Auto mode에서도 바로 진행합니다.

---

## 6. 판단이 모호한 경우

"이 코드 어떻게 고치는 게 좋을까?"처럼 질문 형태지만 암묵적으로 구현을 요청하는 경우:
- 분석·설명으로 답하고, 직접 수정이 필요해지는 시점에 가드를 발동합니다.
- Edit/Write 도구를 호출하려는 순간이 "구현 시작 시점"입니다.
