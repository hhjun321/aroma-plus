# Input Validation Standards

## Table Stakes 밸리데이션

사용자가 별도로 요청하지 않아도 **모든 사용자 입력 필드**에는 다음 밸리데이션을 반드시 포함합니다.

### 1. DB 제약 정합성 (Infrastructure Safety)

DB 에러를 유발하는 입력을 애플리케이션 레이어에서 먼저 차단합니다:

- `varchar(N)` 컬럼 → 최대 N자 제한
- `NOT NULL` 컬럼 → 빈 값 거부
- 숫자 타입 컬럼 → 범위 검증

### 2. 타입 시맨틱 (Semantic Integrity)

필드명이 암시하는 데이터 형태를 검증합니다. 사용자가 포맷을 언급하지 않아도 자동 적용합니다:

| 필드 유형 | 최소 검증 |
|----------|----------|
| `email` | `@` 포함 여부, 기본 이메일 형식 |
| `phone` | 대한민국 전화번호 형식 정규식 / 숫자와 `+` 이외의 문자 불허 (하이픈·공백·괄호 금지) |
| `url` | `http://` 또는 `https://` 시작 |
| `password` | 최소 길이 (서비스 정책에 따라 8자 이상 권장) |

#### `phone` 필드 상세 규칙

허용 형식은 **숫자만** (단, 국제 전화 prefix `+82`에 한해 `+` 허용). 하이픈(`-`), 공백, 괄호 등 구분자는 **절대 허용하지 않습니다.**

```
올바른 예시
  01011112222       — 휴대폰 (국내 번호)
  +821011112222     — 휴대폰 (국제 번호)
  0212345678        — 서울 지역번호 (8자리)
  021234567         — 서울 지역번호 (7자리)
  0311234567        — 경기 지역번호

잘못된 예시
  010-1111-2222     — 하이픈 불허
  +82-10-1111-2222  — 하이픈 불허
  (02) 123-4567     — 괄호·공백·하이픈 불허
  12-34-56          — 미지원 형식
```

정규식: `^(?:\+82|0)(?:10\d{8}|2\d{7,8}|(?:3[1-3]|4[1-4]|5[1-5]|6[1-4])\d{7,8})$`

- `+82` prefix 사용 시 국내 `0`은 제외하고 입력 (예: `+8210...`)
- 필드가 nullable인 경우 빈 값·null은 통과, 값이 있을 때만 위 정규식 적용

### 3. 보안 경계 (Security Boundary)

- 모든 문자열 입력: 합리적인 최대 길이 제한 (DoS 방지)
- 자유 형식 텍스트: 허용 문자셋 또는 길이 제한 중 하나 이상

## "제약 없이 저장하자" 요청의 해석

사용자가 특정 필드에 "제약 없이", "자유롭게", "형식 상관없이" 저장하자고 요청한 경우:

**비즈니스 로직 자유도로 해석**: 특정 포맷 강제 안 함, 정규화 안 함

```
✅ 올바른 해석: 01012345678 / +821012345678 모두 허용 (포맷 정규화 안 함)
❌ 잘못된 해석: DB 길이 제한, 허용 문자 검증, 정규식 검증까지 생략
```

> **phone 필드 예외**: phone은 "형식 상관없이"라고 해도 대한민국 번호 형식 검증(정규식)은 항상 적용합니다.
> 구분자(하이픈, 공백, 괄호)는 어떤 경우에도 허용하지 않습니다.

즉, **저장 포맷의 자유도**와 **입력 무결성 검증**은 별개입니다. 후자는 요청 여부와 무관하게 항상 포함합니다.

모호한 경우 다음과 같이 확인합니다:
> "포맷 정규화를 하지 않는 것으로 이해했습니다. 길이 제한과 허용 문자 검증은 포함하겠습니다. 이와 다른 의도이시면 말씀해주세요."

## 체크리스트

사용자 입력을 받는 API 엔드포인트 구현 완료 후:

- [ ] 각 파라미터에 DB 컬럼 길이와 일치하는 최대 길이 제한이 있는가?
- [ ] 필드 시맨틱에 맞는 형태 검증이 있는가? (email, phone, url 등)
- [ ] 자유 형식 텍스트에 길이 제한 또는 허용 문자셋 제한이 있는가?
